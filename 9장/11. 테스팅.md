# 11. 테스팅
- 개발자라도 테스팅 질문을 받을 수 있다
- 테스팅 질문은 보통 다음 4개중 하나
1. 실생활에서 접하는 객체(ex> 펜)를 테스트하라
2. 소프트웨어 하나를 테스트하라
3. 주어진 함수에 대한 테스트 코드를 작성하라
4. 발생한 이슈에 대한 해결책을 찾아내라

## 면접관이 평가하는 것
- 큰그림을 이해하고 있는가.
- 퍼즐 조각을 제대로 맞추는 방법을 아는가
- 조직화: 문제를 구조적으로 접근하는가
- 실용성: 실제로 적용 가능한 합리적인 테스트인가?

## 실제 세계에서 객체 테스트하기
1. 사용자는 누구인가? 클립의 사용 목적은 무엇인가?
2. 어떤 유즈케이스(use case)가 있나?
3. 한계조건은?
4. 스트레스 조건과 장애 족너은?
5. 테스트는 어떻게 수행할 것인가?

## 소프트웨어 테스팅

소프트웨어 테스팅의 두가지 핵심적 측면

- 작업 테스트 vs. 자동화 테스트
    - 모든것을 자동화 하면 좋겠지만 불가능함
    - 적절한 경우에 두가지를 나눠 쓰는게 좋음
    - 인간과 컴퓨터는 둘 다 테스트 프로세스틔 핵심적 부분

- 블랙박스 테스트 vs. 회이트박스 테스트
    - 소프트웨어 내부를 어디까지 들여다 볼 수 있느냐에 근거
    - 블랙박스: 주어진 그대로 테스트
    - 화이트박스: 내부의 개별 함수들을 프로그램적으로 접근하여 테스트
    - 블랙삭스 트스트 자동화는 가능하긴 하지만 훨씬 더 어려움

테스트 접근법

1. 블랙박스 테스트/화이트박스 테스트 중 어느것 혹은 둘다 해야하는가?
2. 누가 사용할 것인가? 왜 사용하는가?
3. 어떤 유스케이스들이 있나?
4. 한계 조건은?
5. 스트레스 조건과 장애 조건은?
6. 테스트 케이스는? 테스트 실행은 어떻게?

## 함수 테스트

단계별 진행방법: sort(int[] array)를 테스트하라는 문제를 받았다고 하자.

1. 테스트 케이스 정의
    - 정상적인 케이스
    - 극단적인 케이스
    - null 입력, 잘못된 입력
    - 특수한 입력
2. 예상되는 결과를 정의하라
3. 테스트 코드를 작성하라

## 문제 해결에 관한 문제

이미 있는 장애를 어떻게  디버깅하고 해결할 것인지 설명하라는 문제

1. 시나리오를 이해하라
    - 상황을 가장 정확하게 이해할 수 있도록 많은 질문을 던짐
2. 문제를 쪼개라
    - 문제를 테스트 가능한 단위로 분할
3. 구체적이고 관리 가능한 테스트 들을 생성하라
    - 구성요소들 각각에 대한 현실적인 지시사항이 있어야 한다
